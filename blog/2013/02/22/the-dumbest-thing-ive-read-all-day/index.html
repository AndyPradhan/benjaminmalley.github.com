Time to End the Java Hate

Feb 22nd, 2013

I was just beginning to like The Verge and then bam! they published this article in which the author fails to differentiate between Java applets, which were always a terrible idea and the JVM, which continues to be an awesome piece of kit and arrives at the completely erroneous conclusion that Java needs to die.

First, let’s separate Java the language from Java the platform. Without Java the platform, we wouldn’t have Clojure, Scala or the Android SDK (the author admits as much). I don’t think there’s anyone out there in the world advocating we get rid of those, so that clearly needs to stay.

This leaves us with Java the language for which there’s a lot of hate out in the wild. Some of this, clearly, is deserved. Java’s list of features is short. Type erasure was a mistake. Null pointer exceptions were probably a mistake. Changes to the language are slow to be implemented and there’s a growing sense that it’s just not catching up fast enough.

I’ve been doing some Android programming lately and one of the very first things I did was scout around for ways to do it in anything but Java. There are a few options, but I decided that since what I would be spending most of my time with was interacting with the SDK class, I bit the bullet and dug in with Java. And in so doing, I found a kind of beauty. The language is minimal enough that I almost never have to think about how to accomplish something. In Java, there’s usually a single best way to accomplish a particular goal and it’s usually straightforward (although, yes, pity the cases when it is). Others can look at my code and understand at a glance what I am doing.

There are those out there who would say that Scala “replaces” Java. Wikipedia says it was designed to be a “better” Java. Looking at its feature set, I can understand why one would think it has succeeded. But, seriously, look at it: Scala is a huge hulking mess! It looks and feels exactly like a language that is designed to be all things to all people. Semi-colons are optional. The return keyword is optional. Methods can be called infix style.

I don’t think Scala is bad.

Scala supporters claim that their language fixes some of Java’s flaws. Wikipedia asserts that it was designed to be a “better Java.” Looking at the list of features, it ticks every box I could ever want for Java: value-oriented programming, expression-oriented programming, immutable collections, mixins, good concurrency support, even algebraic data types are in there! It’s a damn good list and if I chose programming languages exclusively based on those lists, I’d pick Scala over Java any day of the week.

Sometimes I am, in fact, tempted to pick Scala. But every time I am, there’s one thing that holds me back: the language is a complete mess. I see Scala as something like the anti-Lisp. There’s so much syntax needed to support the many paradigms Semi-colons are optional. The return keyword is optional. For every problem there are sixteen or seventeen ways to approach it. I program in Haskell a lot, so I know how this feels.
